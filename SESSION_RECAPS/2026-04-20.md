## Analyse-Phase — 3 Externe Repos

### 1. polymarket-cli (Polymarket/polymarket-cli, Rust, v0.1.5)

**Was analysiert:**
Offizielles Polymarket CLI-Tool in Rust. Untersucht auf Nutzbarkeit für Auto-Claim.

**Befunde:**
| Funktion | Status | Erklärung |
|----------|--------|-----------|
| `clob balance` | ✅ Funktioniert | Liest USDC-Balance korrekt |
| `wallet show` | ✅ Funktioniert | Zeigt EOA + Proxy-Adresse |
| `data positions` | ✅ Funktioniert | Liest Portfolio |
| `ctf redeem` | ❌ Fehlschlagend | EOA hält keine CTF-Tokens |

**Kern-Erkenntnis (P086):** CTF-Tokens (ERC-1155) liegen im PROXY-Wallet (0x700BC51b),
nicht im EOA (0xd7869A5C). polymarket-cli ist EOA-only designed.

**Beweis via API:**
```bash
GET /positions?user=0x700BC51b&redeemable=true → 13 Ergebnisse
GET /positions?user=0xd7869A5C&redeemable=true → 0 Ergebnisse
```

**Implikation:** Auto-Claim via polymarket-cli nicht möglich. RelayClient (relayer-v2) bleibt
der einzig korrekte Weg. Notification-only (aktuell deployed) ist ausreichende Zwischenlösung.

### 2. TradeSEB/polymarket-copytrading-bot (Python)

**Was analysiert:**
Copy-Trading-Bot in Python, ähnliche Architektur wie KongTradeBot. Untersucht auf verwertbare Patterns.

**Übernommenes Pattern — Exponential Backoff (P089):**

```python
# utils/retry.py — TradeSEB-Source
# 2s → 4s → 8s bei retryable Errors
```

Retryable: network, timeout, ECONNREFUSED, RPC, rate limit, 503/502/504, socket, ankr
Non-retryable: invalid hex address, auth errors, market not found

**Nicht übernommen:**
- Direkter CTF-Contract-Call (ethers.js Ansatz): Funktioniert nicht mit Proxy-Wallet (P086)
- Einzelne Code-Patterns die direkt auf EOA setzen

## Wichtigste Erkenntnisse des Tages

### P086 — Proxy vs EOA Token-Holding (Kern-Erkenntnis)

Das ist die wichtigste Erkenntnis der Session. Alle bisherigen Auto-Claim-Versuche
scheitern an dieser Architektur-Realität:

```
EOA (0xd7869A5C) = Signer, CLOB Owner, PRIVATE_KEY
           ↓
Proxy (0x700BC51b) = hält CTF-Tokens, führt Transaktionen aus
```

CTF-Contract-Calls müssen vom Proxy signiert werden — nicht vom EOA.
Das geht nur via RelayClient (relayer-v2.polymarket.com).

### P087 — ?redeemable=true Endpoint (Operativer Gewinn)

Neu entdeckter Endpoint der State-Detection massiv vereinfacht:

```
GET /positions?user={PROXY}&redeemable=true
→ 1 API-Call für alle resolved Positionen
→ statt N Gamma-API-Calls
```

Test-Ergebnis: Alle 13 redeemable Positionen sind RESOLVED_LOST ($148.71 verloren, $0 claimable).
**redeemable=true bedeutet "Markt resolved" — nicht "Gewinn".**

### P088 + P089 — Lessons aus externen Repos

- polymarket-cli: Nützlich für Portfolio-Inspektion, unbrauchbar für Claims
- Retry-Pattern: Einfacher Exponential-Backoff ist Best-Practice, kein Overkill
